## 二次修复（结合 `gagmate_combined(3).log` 网络日志核对数据契约）

用户提供真实机器日志做参考，核对出上一轮修复仍不充分的根因：

1. **实时曲线空白真正根因**：`ShotRepository` 旧逻辑在每次 shot 快照 `timeInShot < 100` 时 `clearChart()`。固件 `timeInShot` 字段名若不匹配（Gson 解析为 0）→ 每帧清空缓冲区 → 曲线永远空白。已改为**仅 `brewActive` 上升沿清空**，时间轴改用**墙钟时间差兜底**（优先固件 `timeInShot`，否则萃取开始起算秒数）。`appendSensorPoint` 合并为 `appendPoint(time,p,f,t,w)`。

2. **profile 曲线 0 值真正根因（日志实证）**：本机 `GET /api/profile/{id}` **独立端点未被 App 调用**；但 **shot 记录 (`api/shots/{id}`) 稳定内嵌完整 `profile.phases`**，schema 与 `PhaseV3`/`PhaseTarget` 完全一致：
   - `type` 大写 `"FLOW"`/`"PRESSURE"`；`curve` 是**字符串**；首阶段 `target` 常缺 `start`（按 0 起）。
   - 例：`{"target":{"end":1.5,"curve":"EASE_OUT","time":15000},"type":"FLOW",...}`。
   - 这正是 `PhaseV3.toBrewPhase()` 能正确解析的格式 → 之前 REST 之所以 0，是端点没被命中 / 固件不支持独立 detail。
   - 新增 `MachineRepository.fetchProfilePhases(id, name)`：优先 `GET /api/profile/{id}`，**回退最近 shot 内嵌 profile（按 name 匹配）**。详情页 + 仪表盘均改用此聚合入口。

## 纠正：shot 回填方案有误（用户第 4 轮反馈）

用户指出：shot 是**萃取历史快照**，用它当离线 canonical 数据语义错误——萃取后若在机器改了曲线，shot 内嵌的还是旧配方。**正确做法：拿到曲线「当前」profile 详情就落库**，离线读库里的当前数据。

### 改动（撤回 shot 回填，改为 REST 详情落库）
- 撤回 `SyncManager.syncShots()` 的 shot 内嵌回填逻辑（及 `EmbeddedProfile`/`toBrewPhase` 临时 import）。
- `SyncManager.syncProfiles()`：对每台机器 profile 调 `machineRepo.getProfileDetail(mp.id.toString())`，取 `phases` → `toBrewPhase()` → 序列化写本地 `phasesJson`（覆盖 SYNCED，保护 MODIFIED/CONFLICT/LOCAL_ONLY）。新增 profile 直接带 phases 落库，不再写 `"[]"` 等 WS。
- `ProfileRepository` WS→Room 收集器：增加全零保护——若 WS 推送 `phases` 全为 0（target==0 && time<=0.1），**丢弃不写库**，避免覆盖 REST 落库的权威数据。
- `fetchProfilePhases()` 仍为联网实时展示用（REST 详情 → shot 内嵌兜底），但**不再负责落库**。
- 仪表盘 / 详情页的本地库回退保持不变（现已读到正确的当前数据）。

### 编译
